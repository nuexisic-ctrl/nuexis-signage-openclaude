# NuExis System Audit Report

## Executive Summary

An extensive system audit was performed on the NuExis codebase focusing on security posture, scalability limitations, application architecture, and product feature improvements. The application presents a solid multi-tenant foundation using Next.js and Supabase. However, several critical vulnerabilities, severe performance bottlenecks, and architectural shortcuts were identified that pose a significant risk to data integrity, platform stability, and overall operational scalability. 

The most urgent findings (P0) involve the use of service role keys that bypass all Row Level Security (RLS) policies, unauthenticated access to the `workspace-media` storage bucket, and aggressive 5-second polling loops generated by devices which will crash the database at scale. This report prioritizes all identified risks, classifies their severity, and offers explicit remediation steps. Immediate action is required for all "Critical" and "High" severity issues before considering any production deployment.

## Security Findings

### 1. Hardcoded Service Role Key Exposes Data and Bypasses RLS

* **Severity:** Critical
* **Affected files/components:** `app/player/actions.ts`, `app/signup/actions.ts`
* **Description of the issue:** The `SUPABASE_SERVICE_ROLE_KEY` is utilized inappropriately within server actions (such as `getSignedMediaUrl` and team slug uniqueness check). This completely bypasses Row-Level Security (RLS) rules in PostgreSQL, making the system highly vulnerable. While its scope may be restricted to storage URLs in some places, utilizing service role keys outside of strictly controlled backend jobs creates a catastrophic attack surface if parameters are ever mishandled.
* **Potential impact:** Complete tenant isolation breach, potential data exfiltration, deletion of cross-tenant data. 
* **Example attack or failure scenario:** An attacker manipulates the `filePath` parameter in a `getSignedMediaUrl` request. Because the service role bypasses RLS, the backend ignores ownership policies and returns a valid signed URL for another tenant's proprietary files.
* **Exact recommendation to fix it:** Remove all usages of `SUPABASE_SERVICE_ROLE_KEY` from player actions and client-facing endpoints. Transition to using the Supabase anonymous key (`NEXT_PUBLIC_SUPABASE_ANON_KEY`) in combination with strictly scoped PostgreSQL `SECURITY DEFINER` RPCs that handle operations securely inside the database.

### 2. Public Storage Bucket Allows Unauthenticated Data Access

* **Severity:** High
* **Affected files/components:** Supabase Storage (`workspace-media` bucket)
* **Description of the issue:** The `workspace-media` bucket is configured as `public: true`. This allows anyone on the internet with the file path to download the contents of the bucket without authentication.
* **Potential impact:** Complete exposure of all user-uploaded content, leading to data breaches, compliance violations, and intellectual property theft.
* **Example attack or failure scenario:** An attacker enumerates the URL schema (`{team_id}/{timestamp}-{filename}`) or discovers the URLs in HTTP traffic, successfully downloading sensitive corporate video assets without ever authenticating.
* **Exact recommendation to fix it:** 
  1. Configure the `workspace-media` bucket to `public: false`.
  2. Continue utilizing signed URLs with short Time-To-Live (TTL) values, ensuring these URLs are generated by an authenticated mechanism.

### 3. Missing RBAC Enforcement on Mutating Server Actions

* **Severity:** High
* **Affected files/components:** `app/customer/[team_slug]/asset/actions.ts`, `app/customer/[team_slug]/playlists/actions.ts`, `app/customer/[team_slug]/screens/actions.ts`
* **Description of the issue:** While a `role` exists on `profiles` (default `owner`), many server actions are not consistently verifying these roles across all mutations. It appears any authenticated member within a team can execute actions like `deleteAsset`, `deletePlaylist`, or modify devices.
* **Potential impact:** Unauthorized deletion or modification of business-critical assets by non-privileged team members.
* **Example attack or failure scenario:** A standard team member (e.g., an 'editor' or 'viewer') issues a POST request directly to the server action endpoint to delete an entire digital signage playlist, successfully bypassing UI restrictions.
* **Exact recommendation to fix it:** Apply strict Role-Based Access Control (RBAC) to every mutating server action. Ensure `requireOwner` (or an equivalent role-check utility) is called before processing any insertions, updates, or deletions.

### 4. Device Secret Stored as Plaintext

* **Severity:** Medium
* **Affected files/components:** PostgreSQL Database (`devices` table)
* **Description of the issue:** The `secret` column on the `devices` table is stored in plaintext. This secret acts essentially as a password for device authentication.
* **Potential impact:** If the database is compromised, all device authentication tokens are exposed, allowing the attacker to hijack every digital screen managed by the platform.
* **Example attack or failure scenario:** A read-only SQL injection vulnerability or database backup leak exposes the `devices` table. The attacker uses the plaintext secrets to take over all signage displays and broadcast malicious content.
* **Exact recommendation to fix it:** Hash the `secret` column using `pgcrypto` (`crypt()`) upon creation, and compare hashes for authentication, treating device secrets identically to user passwords.

### 5. Inadequate Rate Limiting Coverage on Sensitive Endpoints

* **Severity:** Medium
* **Affected files/components:** `app/customer/[team_slug]/screens/actions.ts` (`updateDeviceLastSeen`)
* **Description of the issue:** While login and signup have rate limiting, other sensitive server actions lack adequate rate-limiting or storage quotas.
* **Potential impact:** Denial of Service (DoS) and potential financial impacts due to unbounded API requests.
* **Example attack or failure scenario:** An attacker script repeatedly calls `updateDeviceLastSeen` or continuously requests upload URLs without pushing files, exhausting database connections and cloud quota limits.
* **Exact recommendation to fix it:** Apply standard rate limiting using Redis across all server actions, not just authentication endpoints. Ensure storage quotas are tracked and enforced for media uploads.

## Scalability & Performance Findings

### 1. Aggressive 5-Second Device Polling Loop

* **Severity:** Critical
* **Affected files/components:** `app/player/page.tsx` (as per `recommendation.md`), `app/player/actions.ts` (`getDeviceState`)
* **Description of the issue:** Devices execute a 5-second polling loop to fetch state updates (`getDeviceState`). 
* **Potential impact:** At a scale of 10,000 devices, this creates 120,000 database requests per minute. This will rapidly exhaust Supabase connection limits, overload the CPU, and create unacceptable latency across the platform.
* **Example failure scenario:** A new enterprise customer connects 5,000 screens. The resulting API and database load causes total service degradation, making the web dashboard completely unresponsive for all users.
* **Exact recommendation to fix it:** 
  1. Remove the active polling loop. 
  2. Implement Supabase Realtime (WebSocket) to push `postgres_changes` on the `devices` table directly to the connected players.
  3. If a fallback polling mechanism is strictly required, increase the interval to 5-15 minutes and implement randomized exponential backoff (Jitter) to prevent thundering herd problems.

### 2. Redundant Realtime Presence and Database Playtime Updates

* **Severity:** High
* **Affected files/components:** `app/player/page.tsx` (as per `recommendation.md`), `app/player/actions.ts`
* **Description of the issue:** Every 60 seconds, every device manually tracks presence, calls `incrementPlaytime(...)`, and calls `sendHeartbeat(...)`. This creates an extreme amount of redundant database writes and WebSocket broadcast messages.
* **Potential impact:** PostgreSQL row contention, massive Write-Ahead Log (WAL) bloat, and exhaustion of Supabase Realtime quota limits.
* **Example failure scenario:** With 5,000 active devices, the database experiences 5,000 `UPDATE` commands per minute on the same tables alongside extreme WebSocket noise, leading to transaction locks and timeouts.
* **Exact recommendation to fix it:**
  1. Remove manual 60s `.track()` calls. Supabase Realtime manages presence implicitly.
  2. Batch playtime updates. Store playtime accrual in device browser memory and flush to the database periodically (e.g., every 15-30 minutes) or upon `visibilitychange`/tab unload via `navigator.sendBeacon`.
  3. Consolidate online/offline status purely via Redis TTLs rather than forcing manual database syncs from the frontend client.

### 3. Full Table Scans Due to Missing Foreign Key Indexes

* **Severity:** Medium
* **Affected files/components:** PostgreSQL Database Schema
* **Description of the issue:** Multiple foreign keys across tables (e.g., `devices`, `playlist_items`, `playlists`, `profiles`) lack associated indexes.
* **Potential impact:** High CPU usage and slow query execution times. Any `CASCADE` deletions or complex `JOIN` queries will result in sequential table scans rather than index lookups.
* **Example failure scenario:** When a user deletes a team or a large playlist, the database must sequentially scan every row in the `devices` and `playlist_items` tables, causing the request to hang and potentially time out.
* **Exact recommendation to fix it:** Run SQL migrations to generate indexes on all foreign key columns.

### 4. Poor Caching and Unpaginated Queries

* **Severity:** Medium
* **Affected files/components:** API endpoints querying assets and playlists
* **Description of the issue:** The system fetches the entire list of assets and playlists without employing cursor or offset pagination.
* **Potential impact:** High bandwidth usage, slow Time-To-First-Byte (TTFB), and increased memory consumption on the client.
* **Example failure scenario:** A user with a library of 10,000 video assets navigates to the dashboard, crashing the browser tab attempting to render the entire dataset simultaneously.
* **Exact recommendation to fix it:** Implement cursor-based pagination for all list queries (e.g., assets, playlists) and utilize edge caching mechanisms via Next.js for largely static elements.

## Architecture & Code Quality Findings

### 1. Lack of Automated Testing

* **Severity:** High
* **Affected files/components:** Codebase-wide
* **Description of the issue:** The project lacks comprehensive unit, integration, and End-to-End (E2E) testing. Initial runs showed a completely empty test suite (or missing test dependencies before intervention).
* **Potential impact:** High risk of regression. Refactoring or adding new features will easily break existing functionality silently.
* **Exact recommendation to fix it:** Establish a robust CI/CD pipeline. Enforce minimum test coverage thresholds utilizing Vitest and React Testing Library (which were recently installed). Write tests for all core business logic, including rate limiters and authentication flows.

### 2. Lack of Transactional Boundaries for Complex Mutations

* **Severity:** Medium
* **Affected files/components:** Playlist Database Queries
* **Description of the issue:** Modifying playlists involves deleting existing items, inserting new items, and updating the playlist name. Currently, if the insert step fails, the previous deletion persists, leaving the playlist permanently empty.
* **Potential impact:** Data loss and inconsistent state resulting from mid-operation failures.
* **Exact recommendation to fix it:** Wrap complex mutations in database transactions. Use Supabase `SECURITY DEFINER` RPCs to encapsulate the Delete/Insert/Update sequence inside a single Postgres transaction block so it safely rolls back on failure.

### 3. Missing React Error Boundaries

* **Severity:** Low
* **Affected files/components:** React Application (`app` directory)
* **Description of the issue:** The React application lacks overarching Error Boundaries.
* **Potential impact:** A single unhandled error in a minor component will unmount the entire React tree, resulting in a blank white screen for the user.
* **Exact recommendation to fix it:** Implement global and component-level Error Boundaries (e.g., using `react-error-boundary`) to catch rendering errors and display a graceful fallback UI.

## Recommended Improvements

1. **Implement Proper RBAC System:** Finalize and enforce the Role-Based Access Control logic utilizing custom claims in JWTs, allowing granular permissions (Owner, Editor, Viewer).
2. **Move to Cursor Pagination:** Replace all bulk table fetches with infinite scroll and cursor-based pagination interfaces to handle massive data sets efficiently.
3. **Robust Observability:** Integrate an Application Performance Monitoring (APM) tool (like Sentry or Datadog) to track client-side errors, API latencies, and database queries in production.
4. **Implement Global Rate-Limiter Middleware:** Move IP-based rate limiting to Next.js Edge Middleware to drop malicious requests before they consume compute resources.
5. **Secure Database Secrets:** Immediately migrate the `devices.secret` storage strategy to utilize `bcrypt` or `pgcrypto`.

## Suggested New Features

1. **Advanced Content Scheduling (Timezone Aware):** Introduce a scheduling engine that allows users to dictate playlist rotations based on specific dates, days of the week, and local device timezones.
2. **Device Grouping and Bulk Operations:** Enable users to tag, group, and execute bulk operations (assign playlist, reboot, change orientation) on multiple screens simultaneously.
3. **Multi-Region CDN for Assets:** Deploy a globally distributed CDN for `workspace-media` to drastically reduce latency and buffering times for screens located far from the primary server.
4. **Offline Resilience Mode for Players:** Utilize IndexedDB and the Service Worker to heavily cache the active playlist so signage displays can survive extended internet outages seamlessly.
5. **Detailed Proof-of-Play Analytics:** Offer a monetization or compliance feature where customers can generate CSV reports proving exactly how many times and for how long a specific media asset was played on screens.

## Quick Wins (Immediate Value)

* Replace service role keys with the anon key and RLS policies in all client paths.
* Lock down the `workspace-media` bucket (`public: false`).
* Remove the `.track()` 60-second presence pulse.
* Add database indexes to the 6 identified foreign keys.
* Configure strict rate limits on all file upload and claiming mechanisms.

## Long-Term Recommendations

* **Migration Strategy:** The presence of numerous rapid database migrations suggests lacking environments. Set up a strict staging environment and a defined GitOps CI/CD pipeline for safe deployment.
* **Database Offloading:** Evaluate moving rapid, ephemeral data (like `last_seen_at` and heartbeat tracking) entirely out of PostgreSQL and completely into Upstash Redis to vastly reduce Postgres write-load at enterprise scale.
* **Compliance:** Adopt SOC2 security principles, including audit logging for all mutating administrative actions to prepare the application for enterprise sales.
