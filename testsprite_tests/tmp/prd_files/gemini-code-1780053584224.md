# Product Requirements Document (PRD) - Security Testing Scope

## 1. Project Overview & Product Functionality
**Name:** NuExis Digital Signage
**Architecture:** Next.js (App Router, Frontend & API) and Supabase (PostgreSQL, Auth, Storage, Row Level Security).
**Product Description:** NuExis is a multi-tenant web application that allows organizations to remotely manage digital signage displays. Users can upload media files, organize them into scheduled playlists, and assign those playlists to specific screens connected to their workspace.

## 2. Core Features & Web Application Pages
The application is organized into several key sections under a tenant-specific routing structure (`/customer/[team_slug]/...`). TestSprite must navigate and evaluate the following pages:

* **Dashboard (`/customer/[team_slug]/dashboard`):** The central hub displaying high-level analytics, active screen counts, device health widgets, and recent activity timelines.
* **Assets Library (`/customer/[team_slug]/asset`):** A media management page featuring an `UploadZone` for adding images/videos, an `AssetBrowser` for filtering and viewing uploaded files, and preview modals.
* **Playlists (`/customer/[team_slug]/playlists`):** The interface where users group media assets into sequences. Users can create, edit, and delete playlists, defining how and when media plays.
* **Screens Management (`/customer/[team_slug]/screens`):** The hardware management interface. Users can pair new screens to their workspace using a pairing code, monitor device uptime/health via a stats grid, assign specific playlists to individual screens, and rename or delete devices.

## 4. Targeted Security Audit Scope
TestSprite is instructed to execute tests across the following attack surfaces, utilizing the provided credentials:

### 4.1 Tenant Isolation & Row Level Security (RLS) Bypass
* **Target:** Supabase PostgreSQL RLS policies and Next.js data fetching logic across all core pages (Dashboard, Assets, Playlists, Screens).
* **Objective:** Ensure the `test` user cannot read, mutate, or delete records belonging to other tenants.
* **Attack Vectors:** Attempt IDOR (Insecure Direct Object Reference) by manipulating UUIDs in API requests. Attempt to access routes belonging to non-existent or administrative slugs (e.g., `/customer/admin/screens`).

### 4.2 Asset & Media Upload Vulnerabilities
* **Target:** `UploadZone` component and Supabase Storage buckets on the Assets page.
* **Objective:** Prevent malicious file execution, MIME-type spoofing, and Stored XSS.
* **Attack Vectors:** Upload disguised files (e.g., an HTML file renamed to `.png`). Upload malicious SVG files containing JavaScript payloads to test for Stored XSS when the image renders in the Asset Preview modals or Dashboard widgets.

### 4.3 Client-Side Routing & API Endpoint Security
* **Target:** Next.js middleware, route handlers (e.g., `/api/player/media-url`), and protected page layouts.
* **Objective:** Prevent unauthorized access to pages and data manipulation.
* **Attack Vectors:** Attempt to access protected routes (like `/customer/test/playlists`) in an unauthenticated state to test middleware redirects. Submit unauthorized POST/DELETE requests to API endpoints to manipulate screens or playlists without a valid session token.
